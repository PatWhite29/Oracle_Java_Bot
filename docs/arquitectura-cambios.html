Documentación de Cambios

Resumen técnico del flujo conversacional del bot, KPI de productividad y actualización del dashboard.

Proyecto: Oracle Java Bot · MyTodoList Fecha: Marzo 2026 Stack: Spring Boot · React · Telegram Bot API
Backend – 1

Modelo ToDoItem.java

Se extendió la entidad JPA para almacenar los datos capturados por el bot y los tiempos necesarios para calcular el KPI.

Campos nuevos

CampoTipoDescripción
titleStringTítulo limpio de la tarea (separado de description)
assigneeStringNombre del responsable ingresado por el usuario
complexityStringComplejidad normalizada: Low, Medium o High
startTimeOffsetDateTimeMomento en que se creó / inició la tarea
endTimeOffsetDateTimeMomento en que la tarea se marcó como completada

KPI de Productividad @Transient

El método getProductivityKpi() se calcula en tiempo de ejecución y no se guarda en la base de datos. Se expone en el JSON de la API para que el cliente lo consuma.

KPI = (peso_complejidad × 60) / minutos_trabajados

Peso: High = 3 · Medium = 2 · Low = 1 · Tiempo mínimo: 1 min

Si la tarea no está completada todavía, usa OffsetDateTime.now() como endTime provisional para mostrar un KPI en progreso.

Backend – 2

Servicio ToDoItemService.java

Método applyTaskDefaults()

Se invoca en cada operación de escritura (add y update). Garantiza integridad de datos:

  • Si no hay title, usa description como respaldo y viceversa.
  • Si no hay assignee, queda como "Unassigned".
  • Si no hay complexity, queda como "Medium".
  • Si no hay startTime, se fija igual a creation_ts.
  • Si la tarea está completada y no tiene endTime, se fija en now().
  • Si la tarea se revierte a pendiente (Undo), se borra endTime.

Método normalizeComplexity()

Acepta variaciones en español e inglés (Alta, High, Baja, Low) y las unifica al formato estándar.

Backend – 3

Flujo Conversacional del Bot

Antes del cambio, cualquier mensaje libre se convertía en tarea sin datos adicionales. Ahora el bot guía al usuario paso a paso usando un estado por conversación.

Clase BotConversationState.java

Almacena el estado de cada chat activo en un ConcurrentHashMap<Long, BotConversationState> (clave = chatId). Los estados posibles son:

IDLE → WAITING_TITLE → WAITING_ASSIGNEE → WAITING_COMPLEXITY → IDLE

Flujo completo desde Telegram

1
Usuario envía /start Bot responde con mensaje de bienvenida y pide el título en formato: TITULO: <nombre de la tarea>. Estado pasa a WAITING_TITLE.
2
Usuario envía TITULO: Hacer video demo Bot extrae el valor, lo guarda en estado y avanza a WAITING_ASSIGNEE. Pide formato: ASIGNADO: <nombre>.
3
Usuario envía ASIGNADO: Ana Torres Bot guarda el asignado y avanza a WAITING_COMPLEXITY. Pide formato: COMPLEJIDAD: Low | Medium | High.
4
Usuario envía COMPLEJIDAD: High Bot crea la tarea en BD con todos los campos, fija startTime = now() y responde con resumen. Estado vuelve a IDLE.
✓
Tarea completada (ID-DONE) Bot fija endTime, calcula el KPI y responde con el valor numérico de productividad.

Validación de formato

Si el usuario escribe sin el prefijo correcto (ej. escribe solo "Ana" en vez de "ASIGNADO: Ana"), el bot responde con el mensaje de error y no avanza al siguiente paso.

Formato inválido. Usa exactamente: ASIGNADO: <nombre del responsable>

Archivos modificados

  • BotConversationState.java — nuevo archivo, manejo de estado por chat
  • BotActions.java — nuevos métodos: fnCollectTaskData(), handleTitleStep(), handleAssigneeStep(), handleComplexityStep()
  • BotMessages.java — nuevos mensajes de instrucción y error para cada paso
  • ToDoItemBotController.java — se agregó actions.fnCollectTaskData() al pipeline del bot
Frontend – 1

Sincronización Automática con el Backend

El dashboard se actualiza solo cada 5 segundos usando setInterval dentro de un useEffect. No requiere recargar la página para ver tareas nuevas del bot.

Función mapApiTask(item)

Transforma cada item devuelto por GET /todolist al formato que entiende el dashboard:

{
  id:         "api-{ID}",
  apiId:      {ID},        // para PUT al completar
  title:      item.title || item.description,
  assignee:   item.assignee || "Unassigned",
  complexity: item.complexity || "Medium",
  status:     item.done ? "Completed" : "Pending",
  done:       item.done,
  source:     "api",       // distingue de tareas locales del formulario
  createdAt:  item.creation_ts
}

Función mergeTasksByTitle()

Combina 3 fuentes sin duplicados, priorizando en este orden:

  1. Tareas del backend (API)
  2. Tareas creadas con el formulario del dashboard (locales)
  3. Datos mock de demo (fallback visual)

La deduplicación es por título normalizado (minúsculas, sin espacios extra), descartando el segundo si ya existe el primero.

Frontend – 2

Formulario "Create Task"

El formulario recolecta título, asignado y complejidad. Se eliminó el campo de horas estimadas.

Campos actuales

CampoValidación
Task titleRequerido, no vacío
AssigneeRequerido, no vacío
ComplexitySelect: Low / Medium / High (default Medium)

Las tareas creadas desde el formulario se guardan en estado local React (localPendingTasks) con source: "local" para distinguirlas de las del backend.

Frontend – 3

Completar Tarea

La función handleCompleteTask(taskId) tiene comportamiento diferente según el origen de la tarea:

OrigenComportamiento
API (source: "api") Hace PUT /todolist/{id} con done: true. El backend fija endTime y calcula el KPI. Actualiza estado local de apiTasks.
Local (source: "local") Mueve la tarea de localPendingTasks a localCompletedTasks sin llamar al backend.

Resumen

Flujo End-to-End

[Telegram]
  /start
    → Bot pide: TITULO: <título>
    → Bot pide: ASIGNADO: <nombre>
    → Bot pide: COMPLEJIDAD: Low | Medium | High
    → Tarea guardada en BD con startTime = now()

[Dashboard — cada 5 seg]
  GET /todolist
    → Tarea nueva aparece en "Pending Tasks"
    → Muestra: Título, Asignado, Complejidad

[Dashboard — botón Complete]
  PUT /todolist/{id} { done: true }
    → Backend fija endTime = now()
    → KPI = (peso × 60) / minutos_trabajados
    → Tarea pasa a "Completed Tasks"

[Telegram — ID-DONE]
  → Mismo efecto que Complete en dashboard
  → Bot responde con valor numérico del KPI
